## Rationale notes

- **D2 vs. alternatives.** The user phrased this as "main = HEAD, branches merge back like golusoris". The chosen option (merge into master, set master as default) matches the golusoris model directly and keeps `upstream/master` usable for future Netflix syncs. The rejected option (rename sycl → main) would lose the historical meaning of "master" vs "sycl" and require retagging everywhere.
- **D6 vs. `%.10f` default.** The user explicitly called out rounding breaking comparisons with the Python test suite. `%.17g` is the IEEE-guaranteed shortest representation that round-trips losslessly — any numerically equal doubles produce bit-identical strings, enabling exact diffing across backends. Fixed `%.10f` would still truncate in some cases; `%.17g` is the mathematically correct answer.
- **D10 keyless OIDC.** Eliminates key-management operational burden. Sigstore's transparency log provides non-repudiation without private keys. Standard for modern OSS projects.
- **D12 JPL-C-STD inclusion.** User explicitly added this after the initial plan listed only Power of 10. JPL Institutional Coding Standard for C is the superset that Power of 10 is Rule 1 of — full 31 rules codify things Power of 10 alone doesn't (compiler strictness levels, banned functions, verification requirements).
- **D13 full distro matrix.** User chose the most comprehensive option. This is more work up-front but catches portability issues (musl vs glibc, different pkg-config behavior, clang vs gcc defaults) that only surface in production environments.
- **D15 full CI matrix.** Same reasoning — user favors coverage over speed. PR runtime will be ~5× vs single-platform but catches OS-specific bugs.
- **D20 all four capabilities.** User declined to narrow scope; the four capabilities partition cleanly (C1/C2/C3 share the inference runtime, C4 is dev-only tooling), so "all four" doesn't multiply complexity the way an unfocused scope would. C4 explicitly does NOT link into libvmaf — it lives in `dev-llm/` and only surfaces through `.claude/skills/`.
- **D21 PyTorch + Lightning.** Mainstream, mature ONNX export, Lightning gives free DDP / mixed-precision / checkpointing. TF/Keras rejected to avoid two ML frameworks in the dev tree; JAX rejected for second-class ONNX export; ggml rejected because it can't train.
- **D22 ONNX Runtime.** Single runtime spans all four GPU backends via execution providers — matches our existing CPU/CUDA/SYCL/HIP backend matrix one-to-one. Avoids forking inference paths per backend. ORT enforces sandboxed graph execution, which we further restrict via an operator allowlist.
- **D23 all four surfaces.** They're not duplicates: C API is the single mechanism; CLI and ffmpeg-filter are thin wrappers around it; `vmaf-train` is the *training* surface (separate concern from inference). Skipping any of them would leave a real consumer (libvmaf C clients, vmaf CLI users, ffmpeg pipelines, model authors) without a path.
- **D26 workspace relocation vs. alternatives.** Options were: (a) leave at repo root + document, (b) move to `ai/workspace/`, (c) move to `python/vmaf/workspace/` (chosen), (d) delete and auto-create. (a) was too noisy given the fork's root is otherwise clean. (b) conflated the *classic* SVM-based harness with the *fork's* tiny-AI training (different frameworks, different ownership). (c) co-locates the scratch with the only code that uses it and keeps tiny-AI strictly under `ai/`. (d) was rejected because the empty-`.gitignore` placeholders document the subdir contract; losing them lowers discoverability. The `VMAF_WORKSPACE` env override preserves the ability to point at a big-disk mount (the main reason someone would want the old repo-root path).
- **D27 non-conservative + experimental features clarification.** The user first wrote "don't make the img pins conservative... use newer versions... especially cuda and even use experimental features (thats where the speed is)", then clarified "with experimental i meant features not version so it should be >13.2 on cuda". So "experimental" = toolchain feature flags on *stable* CUDA ≥13.2 / *stable* DPC++ (relaxed-constexpr, extended-lambda, sycl-unnamed-lambda, per-kernel device-code split, sm_120 Blackwell gencode). Not preview / beta CUDA branches, which would sacrifice reproducibility without a matching speed win. This gets us the GPU perf without the instability.
- **D28 ADR rule.** Implicit operating norm now explicit. The risk this mitigates: a Claude session makes a decision (e.g., directory move, CI gate change, dependency swap), commits it, the session ends, and the rationale is recoverable only from the commit message — which is fine until the commit message is itself a summary that omits the *alternatives considered*. The ADR row preserves "we chose X over Y because Z" in a single auditable log.
- **D29–D34 repo-root cleanup family.** The user's complaint ("so much shit in it that wasnt just cleaned up restructured by lazy netflix folk") captured the principle: the repo root should only contain surfaces that an external consumer sees first. Everything else moves under the subtree that owns it. `resource/` and `matlab/` are consumed only by the Python harness → `python/vmaf/`. `BENCHMARKS.md` is documentation → `docs/`. `unittest` is a script → `scripts/`. `codeql-config.yml` is a GitHub-specific config → `.github/`. `patches/` was a 1-file leftover duplicating `ffmpeg-patches/` → deleted. After this, repo root holds: `README.md`, `LICENSE`, `CONTRIBUTING.md`, `SECURITY.md`, `CLAUDE.md`, `AGENTS.md`, `Makefile`, `meson.build`, `Dockerfile`, `.clang-{format,tidy}`, plus top-level subtrees (`libvmaf/`, `python/`, `ai/`, `mcp-server/`, `docs/`, `scripts/`, `testdata/`, `model/`, `ffmpeg-patches/`, `docker/`, `.github/`, `.vscode/`, `.claude/`). Rejected alternative: leave everything where upstream had it and just add `.gitignore` — that doesn't fix the discoverability problem for new contributors who have to read the tree to know what's live.
- **D35 hooks schema.** The IDE surfaced a hard parse error ("Expected array, but received undefined") because Claude Code's hooks schema changed from the flat `{matcher, command}` shape to a nested `{matcher, hooks: [{type: "command", command}]}` shape. The fork had the old shape. No alternative: either update to the current schema or ship a broken settings file that silently drops the permission allowlist and every registered hook.
- **D39 runtime op-walk + registry pull-forward.** Alternatives weighed: (1) leave the allowlist Python-side only — rejected, Python's `check-ops` runs at export time but the real threat surface is untrusted `--tiny-model` paths loaded by a shipped `vmaf` binary; (2) add `libprotobuf-c` as a meson subproject — rejected, disproportionate dependency surface for one field scan; a 150-LOC field scanner is auditable, has no tag expansion, and doesn't need arbitrary message support; (3) rely on ORT's public graph-introspection API — rejected, ONNX Runtime 1.22 C API does not expose per-node op-type iteration in the stable surface; only training API + experimental surfaces have it; (4) defer the walk to Wave 1 follow-up and ship the smoke pipeline first — rejected, any positive-path shipping widens the attack surface (more `.onnx` consumers) before the negative-path gate is in place. On registry pull-forward: Wave 1 adds 7+ distinct checkpoints across sub-tracks 2.1–2.4; without a single-source-of-truth manifest now, every sub-track would bolt on its own ad-hoc provenance and the Sigstore-signing step would have nothing to hash against.
- **D40 multi-input API vs. concat-along-channel.** LPIPS is structurally two backbone forward-passes over `ref` and `dist` whose activations are then differenced per-layer. Expressing that as a single-input ONNX with `[1,6,H,W]` (ref channels 0–2, dist channels 3–5) would force the exporter to duplicate the SqueezeNet backbone (once operating on channels 0–2, once on 3–5) and *then* re-implement the weight sharing at export time, producing a graph ~2× larger with no runtime benefit. The two-input form lets ORT's optimiser see identical sub-graphs and share them if desired. The cost of the API extension is bounded — `VmafDnnInput`/`VmafDnnOutput` descriptors plus an ORT `Run` call that already accepts arrays of inputs — and pays off again immediately for MobileSal (single-input RGB but still needs the ImageNet helper) and TransNet V2 (single-input 3D tensor over a sliding window). The 2-PR split preserves bisectability: PR A can be reverted without touching any extractor, and PR B's surface is strictly "add a new `.c` + registry row" once the API is in.
- **D36 Wave-1 breadth vs. "ship what's there first".** The alternative was to train + ship only baseline C1/C2/C3 checkpoints and defer everything else. Rejected because (a) the user explicitly asked to "blow the shit out of the tinyai topic" — a narrow checkpoint-only wave would not honor that; (b) LPIPS / saliency / per-shot-CRF / TransNet are all tiny (< 3M params each) and share the existing ORT infrastructure, so marginal cost is low once baselines are proven; (c) the encoder-side work (10-bit `vmaf_pre`, `vmaf_post`, optional FastDVDnet) needs the same allowlist + tensor-I/O extensions as the scoring-side work, so splitting them across waves duplicates infrastructure changes. The `Loop`/`If` allowlist expansion is the single largest risk: mitigated by the bounded-iteration guard (published models always export bounded loops; unbounded loops are the adversarial case we reject at load time). MCP `describe_worst_frames` is intentionally the last-priority item since it's a debugging affordance, not on the critical scoring path.
