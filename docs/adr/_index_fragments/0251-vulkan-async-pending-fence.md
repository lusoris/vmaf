| [ADR-0251](0251-vulkan-async-pending-fence.md) | T7-29 part 4 — Vulkan VkImage import v2 async pending-fence model. Replaces [ADR-0186](0186-vulkan-image-import-impl.md)'s deliberately synchronous v1 design (record/submit/wait the fence in-call) with a per-frame fence ring + deferred drain. Lawrence's 2026-04-30 profile of the FFmpeg `libvmaf_vulkan` filter (Issue #239) confirmed v1's in-call wait serialises CPU and GPU work — exactly the bottleneck the parent ADR predicted. v2 records and submits per import but does not wait; `vmaf_vulkan_import_image()` records to slot `frame_index % ring_size` and returns immediately (back-pressure: if the slot was already in flight from a prior frame, the call waits that prior fence first); `vmaf_vulkan_wait_compute()` blocks on every outstanding fence in submission order and is the natural drain before `vmaf_vulkan_state_build_pictures()` reads back host mappings; `vmaf_vulkan_state_free()` drains the ring before destroying any handle. New public field `VmafVulkanConfiguration.max_outstanding_frames` (0 selects the canonical default `4`; values clamp to `[1, VMAF_VULKAN_RING_MAX]`); readback via `vmaf_vulkan_state_max_outstanding_frames()`. Staging-buffer footprint grows `1× → max_outstanding_frames ×` per direction (ref+dis) — for 1080p 8-bit Y plane ~16 MB host-visible per state. ABI preserved (the ring is fully internal); the FFmpeg patch in `ffmpeg-patches/0006-libvmaf-add-libvmaf-vulkan-filter.patch` needs no signature change. Cross-backend gate keeps `places=4` — async submission does not change which bytes the staging buffer receives, only when the host can read them. Status flips Accepted on v2 wall-clock ≤ 0.7× v1 measured on real hardware (lavapipe's single-threaded software submit may mask the gain).
