| [ADR-0316](0316-cli-parse-long-only-error-fix.md) | `cli_parse` `error()` long-only-option fix (follow-up to [ADR-0311](0311-libfuzzer-harness-expansion.md) / PR #408). Three handlers in `libvmaf/tools/cli_parse.c` for `--threads` / `--subsample` / `--cpumask` were calling `parse_unsigned(optarg, 't' / 's' / 'c', argv[0])` with a synthesised short-option char. Those chars do not appear in `long_opts[]` (the options are long-only), so any non-numeric value (`--threads abc`, `--th=foo`) tripped `error()`'s `assert(long_opts[n].name)` and crashed the binary with `SIGABRT`. Fix passes the long-only enum value (`ARG_THREADS` / `ARG_SUBSAMPLE` / `ARG_CPUMASK`) instead — `error()`'s `< 256` branch already handles long-only correctly, bringing the three call-sites into parity with the seven sibling handlers (`ARG_GPUMASK`, `ARG_FRAME_*`, `ARG_*_DEVICE`, `ARG_TINY_THREADS`). Promotes the parked reproducer `cli_threads_abbrev_assert.argv` from `cli_parse_known_crashes/` to `cli_parse_corpus/` and removes the `known_assert_in_input` early-reject filter from `fuzz_cli_parse.c`. Adds POSIX-only fork()-based regression test `libvmaf/test/test_cli_parse_long_only_args.c`. | Proposed | cli, security, fuzzing, fork-local |
